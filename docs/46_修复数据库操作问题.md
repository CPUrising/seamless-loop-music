## 当前项目状态
- 相关功能的现有代码情况：
  1. **表结构不一致**：`BaseRepository` 和 `DatabaseHelper` 对 `PlaylistItems` 表的定义不同
     - `BaseRepository`：有自增 `Id` 主键 (`Id INTEGER PRIMARY KEY AUTOINCREMENT`)，无唯一约束
     - `DatabaseHelper`：主键是 `(PlaylistId, SongId)` (`PRIMARY KEY(PlaylistId, SongId)`)，无自增 `Id`
     - 这可能导致运行时错误，因为两者依赖的表结构不同
  
  2. **空方法**：
     - `DatabaseHelper.UpdatePlaylistsSortOrder()` 和 `UpdateTracksSortOrder()` 是空的（第240-241行）
     - `PlaylistRepository.BulkSaveTracksToPlaylist()` 是空的（第149-164行）
  
  3. **逻辑问题**：
     - `TrackRepository.Save()` 方法（第142-153行）：插入 `LoopPoints` 和 `UserRatings` 时使用 `INSERT OR IGNORE`
     - 如果记录已存在，不会更新，只会忽略
     - 建议改为 `ON CONFLICT(TrackId) DO UPDATE`（像 `BulkInsert` 那样）
  
  4. **异常处理问题**：
     - `BaseRepository` 第28行：空的 `catch { }` 会吞掉所有异常，不利于调试
  
  5. **接口设计问题**：
     - `IDatabaseHelper` 接口有两个 `AddPlaylist` 方法，语义混淆：
       - `AddPlaylist(string name, ...)` - 添加播放列表
       - `AddPlaylist(int playlistId, string folderPath)` - 添加文件夹到播放列表

- 可复用的逻辑：
  - `TrackRepository.BulkInsert()` 中的 `ON CONFLICT DO UPDATE` 模式可以复用于修复 `Save()` 方法
  - `PlaylistRepository.UpdateSortOrder()` 和 `UpdateTracksSortOrder()` 的实现可以复用到 `DatabaseHelper`
  - `PlaylistRepository` 中的排序更新逻辑已经是完整的，可以直接参考
  - `PlaylistManagerService.RefreshPlaylistAsync()` 中的文件清理逻辑（第224-240行）可以参考用于全局清理

## 缺少的清理与同步机制
- **现有清理逻辑**：
  - ✅ 文件夹关联歌单：`PlaylistManagerService.RefreshPlaylistAsync()` 会在刷新时移除消失的歌曲（第302-309行）
  - ✅ 歌单重复合并：`DatabaseHelper.ApplyMigrations()` 会合并同名歌单
  - ❌ 普通虚拟歌单：不会自动清理已删除的物理文件引用
  - ❌ 全局音乐库：没有自动检测并删除物理文件已消失的数据库记录

- **缺少的清理机制**：
  1. **全局清理**：启动时或定期扫描，删除物理文件已不存在的 Tracks 记录（以及关联的 LoopPoints、UserRatings，通过 ON DELETE CASCADE 自动删除）
  2. **虚拟歌单清理**：播放时检测文件是否存在，或定期清理失效引用
  3. **自动触发**：建议在应用启动时执行一次全局清理，确保数据库与物理状态同步

- **注意**：同文件不同 TotalSamples 是正常情况（如原曲和剪辑版），不应视为重复记录，无需清理

## 可选实现方案

## 实现计划
### 第一阶段：修复现有问题
1. **统一表结构**：决定 `PlaylistItems` 表采用哪种结构（建议采用 `DatabaseHelper` 的复合主键方案，删除 `BaseRepository` 中的自增 `Id`）
2. **补全空方法**：
   - 实现 `DatabaseHelper.UpdatePlaylistsSortOrder()` 参考 `PlaylistRepository.UpdateSortOrder()`
   - 实现 `DatabaseHelper.UpdateTracksSortOrder()` 参考 `PlaylistRepository.UpdateTracksSortOrder()`
   - 实现或删除 `PlaylistRepository.BulkSaveTracksToPlaylist()`
3. **修复逻辑错误**：修改 `TrackRepository.Save()` 中的 `INSERT OR IGNORE` 为 `ON CONFLICT(TrackId) DO UPDATE`
4. **改进异常处理**：在 `BaseRepository` 中至少记录异常日志，而不是空的 `catch { }`
5. **清理接口**：重命名 `IDatabaseHelper` 中的第二个 `AddPlaylist(int playlistId, string folderPath)` 为 `AddFolderToPlaylist()`

### 第二阶段：添加清理与同步机制
6. **全局清理方法**：在 `DatabaseHelper` 中添加 `CleanupMissingFiles()` 方法：
   - 获取所有 Tracks 记录
   - 检查物理文件是否存在
   - 删除不存在的文件记录（依赖 ON DELETE CASCADE 自动清理关联表）
   - **注意**：同文件不同 TotalSamples 是正常情况，不应视为重复
7. **启动时自动清理**：在 `App.xaml.cs` 或 `PlayerService` 初始化时调用清理方法
8. **虚拟歌单保护**：在播放前检查文件是否存在，不存在则跳过并提示用户
